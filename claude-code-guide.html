モデル＆工数 完全ガイド

Claude Codeは モデル × 工数（Effort） の2軸で性能とコストが決まる。 シーンに応じた切替で 月額コストを1/5 に抑えながら品質を維持できる。

モデル 入力 ($/MTok) 出力 ($/MTok) 速度 主用途
Opus 4.7$15$7550-70 t/s最高品質。複雑な設計・長文編集
Opus 4.7 1M$15→$30 ※$75→$150 ※50-70 t/s1M context版。大型HTML全体読込
Sonnet 4.6$3$1580-100 t/sバランス型。日常実装の主力
Haiku 4.5$1$5150-200 t/s並列エージェント・grep的探索
Opus 4.6 レガシー$15$75通常〜高速 ✨/fast 対応

※ 200K超の入力時は単価2倍。
Opusは Sonnet の 5倍コスト、Haiku の 15倍コスト。Sonnetで足りる作業はSonnetに寄せる。

工数 思考トークン目安 速度 用途

③ /fast 高速モードの正体

使うとき：Opus品質が必要だが速度も欲しい時（大型HTMLの連続Edit等）

⑤ トークン節約Tips（実戦則）

同一ファイルEditは1ターンに集約
複数箇所の修正は old_string を大きく取って一括置換。Edit連打はターン数=トークン消費。
Grep優先・全Read禁止
100KB超ファイルの全Readはトークン爆食い。Grep output_mode:"content" head_limit:20 で局所抽出。
/compact をこまめに打つ
15往復超 or 重コマンド後に実行。会話履歴を圧縮して次ターンの入力単価を抑える。
タスクごと新セッション
無関係な新タスクへの切替時は新セッション。コンテキスト引き継ぎは精度低下＋コスト増の両損。
調査はサブエージェントへ
codebase探索・並列分析はAgent toolで分離。メインのコンテキストを枯渇させない。
/cost でキャッシュ比率確認
セッション長期化時に実行。キャッシュヒット率が下がっていたら新セッション推奨サイン。
編集ロック宣言で並列衝突防止
大型HTML編集前に python claim_edit.py claim --target <file>。並列セッションのcommit巻込み事故を防ぐ。
スクリーンショットは最後の手段
画像はテキストの4-10倍トークン消費。構造確認は preview_eval + JS queryで代替する。
Git/GitHub連携で詰まるポイント

Claude Codeは 複数セッション・worktreeで並列作業 するため、Gitの分岐・コンフリクト事故が起きやすい。 実戦で遭遇した落とし穴と対処法をまとめる。

① HEAD と origin/main の「分岐」とは何か

分岐 = 同じ祖先から枝分かれして、両側がそれぞれ別のコミットを積んでいる状態。 この状態で git push しても fast-forward できず、Git は「上書きできません」と拒否する。
┌── a ── b ── c ── … (origin/main 側に58コミット積まれた) 共通の祖先 ────────┤ └── x ── y ── z (ローカルHEAD側に5コミット積まれた)
典型的な発生原因： 別のClaude Codeセッション（別worktree）が origin/main に直接commit & pushしていた一方、 このセッションは古いベースから始まったため、その間の進化を取り込まなかった。

② 分岐したときの対処法（3択）

A: 純粋な貢献分のみ抽出して載せ直す（推奨）
自分の作業ブランチで何が変わったかを見極め、origin/main の最新を取得 → そこに自分の差分だけを手動適用 → 1コミットでpush。 相手側の進化（別セッションの作業）を温存できる。 
git branch backup-detached HEAD                # 念のためバックアップ
git fetch origin main
git checkout -B tmp-merge origin/main          # 最新ベースに乗せ替え
# ここで自分の変更を手動で再適用（必要箇所のみ）
git add -A && git commit -m "feat: ..."
git push origin tmp-merge:main                 # main へ反映
B: cherry-pick（小規模な分岐向け）
自分の数コミットを git cherry-pick で origin/main 上に乗せ直す。 ただし 既に同じ内容が origin/main に入っているとコンフリクトするため、大規模分岐では避ける。 
git checkout -B tmp origin/main
git cherry-pick <commit1> <commit2> …
git push origin tmp:main
C: force-push（最終手段／原則禁止）
git push -f origin HEAD:main で自分の歴史を上書き。 origin/main の他セッションの作業が全消失するため、 自分以外がpushしていないと確信できる場合のみ。GH Pages 配信元なら配信内容も巻き戻る。
ルール：main/master への force-push は受験者の明示許可がない限り絶対やらない。

③ 分岐を起こさない予防策

作業開始前に必ず fetch
新セッション開始時・worktree作成時は git fetch origin → git log origin/main HEAD --oneline で分岐有無を確認。
編集ロック宣言（並列セッション）
大型HTML編集前に python claim_edit.py claim --target <file> でロック宣言。並列セッションの commit 巻き込み事故を防ぐ。
こまめに小さくpush
1セッションで5コミット以上溜めない。pushが遅れるほど他セッションとの分岐リスクが累積する。
detached HEAD で長居しない
worktreeのcheckoutが detached HEAD のまま放置されると、後で「どのブランチに反映するか」判断が難しくなる。作業前に名前付きブランチ化。
バックアップブランチを切る
rebase / cherry-pick / 強制操作の前に git branch backup-YYYY-MM-DD HEAD。事故時に1コマンドで戻れる。
push前の分岐状況確認
git rev-list --count HEAD ^origin/main で「自分が何コミット先行しているか」、 git rev-list --count origin/main ^HEAD で「相手が何コミット先行しているか」を確認。両方0以外＝分岐。

④ 実戦事例: 2026-05-08 hoki-wiki §6.0 追加時

状況：detached HEAD で5コミット積んだ後、push しようとしたら origin/main が58コミット先行していた。

判断：cherry-pick だと QuickReview 等で origin/main と二重実装になりコンフリクト多発。 → A案（純粋な貢献分のみ抽出）を採用。

手順：
  1. git branch backup-detached-2026-05-08 HEAD で歴史を保全
  2. git checkout -B tmp-merge-demand-page origin/main で最新ベースに乗り替え
  3. 新規ファイル（DemandKwhKisoPage関数 / sec06.0データ / .goal-question CSS）のみを手動適用
  4. python build-hoki-wiki.py で再ビルド
  5. 1コミットにまとめて git push origin tmp-merge-demand-page:main
結果：origin/main の既存58コミット（4段階UI・検索・他ページ追加）を温存しつつ、自分の貢献を反映。所要約2分。
